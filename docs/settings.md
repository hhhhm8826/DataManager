# DataManager Settings Guide

## First Launch Examples

When `settings.v2.json` does not exist, DataManager copies its bundled `PROTO`,
`EXCEL`, `JSON`, `CODE`, and `PROTOC` examples into a writable
`example-workspace` under the application data directory. The first settings
file points every workspace root, four existing code outputs, and
`protoc.exe` to that copy.

The initialization runs only when the settings file is missing. Existing
settings and an existing example workspace are never overwritten, so edits
made while evaluating the examples remain intact. Configure different paths
from **설정** when starting a real project.

## Workspace Paths

Open **설정** and configure absolute Windows paths for:

| Field            | Purpose                                                              |
| ---------------- | -------------------------------------------------------------------- |
| Proto 루트       | Top-level `{Name}Table.proto` and `{Name}EnumType.proto` files       |
| Excel 루트       | Generated and imported `.xlsx` workbooks                             |
| JSON 루트        | Message-level JSON exports                                           |
| protoc 실행 파일 | A real `protoc.exe`; `--version` must return `libprotoc major.minor` |

Paths may contain Unicode, spaces, and long segments. Native commands reject
relative paths, traversal, junction/symlink escapes, and files outside these
configured roots.

## Code Outputs

Add one output directory per required target. The supported language IDs are:

| UI label   | Settings ID | Extra executable                       |
| ---------- | ----------- | -------------------------------------- |
| C++        | `cpp`       | None                                   |
| C#         | `csharp`    | None                                   |
| Java       | `java`      | None                                   |
| Python     | `python`    | None                                   |
| Go         | `go`        | `protoc-gen-go`                        |
| Rust       | `rust`      | None; bundled experimental upb codegen |
| Ruby       | `ruby`      | None                                   |
| PHP        | `php`       | None                                   |
| Unreal C++ | `unreal`    | None; generated by DataManager         |

The legacy ID `golang` remains readable and maps to Go. New settings use `go`.
The Go plugin may be next to `protoc.exe` or on PATH. Rust uses the bundled
`--rust_out` generator with
`--rust_opt=experimental-codegen=enabled,kernel=upb`. Each protoc or Unreal run
stages a complete output directory and replaces the configured target only
after successful generation.

The committed rewrite example set uses all eight generators and can be
reproduced with `pnpm fixtures:rewrite`. The native E2E workspace configures a
Rust output directory and verifies an actual `.rs` artifact.

## Diagram Settings

File colors are keyed by exact Proto filename. Saved node coordinates, viewport,
named layouts, and the hub relation threshold are project data stored in
`.datamanager/workspace.json`. The visible hub threshold is an integer from 1
through 50 and defaults to 5. The deprecated `maxNodesPerColumn` settings v2
field remains readable for compatibility but is not an editable layout control.

## Legacy Import

When `config.json` is discovered, review the dry-run table before importing.
Relative legacy paths are resolved from the legacy config directory, not from
the new executable. Import is one-time and does not edit the legacy file or any
workspace data. See `docs/migration.md` for rollback details.
